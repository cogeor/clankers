## Spec: LLM-Driven Synthetic Trajectory Engine for Clankers

### 1) Goal and Motivation

Cold-start RL in robotics simulation fails due to sparse rewards, exploration
walls, and long-horizon credit assignment.  This module bootstraps learning by
using an LLM as a **structured trajectory proposer**: given a scene description
and task goal, the LLM emits a parameterized skill plan that the Clankers
engine validates in Rapier3D physics, records as MCAP episodes, and packages
into an offline dataset for behavior cloning or warm-start RL.

**Key idea:** Replace human demonstrations with LLM-proposed synthetic
demonstrations, validated and augmented at scale in simulation.

**Separation of concerns (Clankers principle):**
* **Rust engine** — physics, scene management, low-level stepping, MCAP
  recording, rendering/viz, domain randomization
* **Python layer** — reward computation, dataset packaging, training loops,
  evaluation, IK/planning/validation, and all datagen orchestration
* **LLM** — plan proposal only; never executes directly

---

### 2) Scope

**Input:**
1. `SceneSpec` — robot config, object placements, workspace bounds, constraints
   (derived from existing `SceneBuilder` + `ObjectConfig` + `RobotConfig`)
2. `TaskSpec` — natural language goal + structured success criteria + reward
   hint (maps to existing `RewardFunction` / `TerminationFn` on Python side)

**Output:**
A validated MCAP dataset of trajectories containing:
* `(observation_t, action_t, observation_{t+1}, terminated, truncated, info)`
  tuples matching the existing `StepResult` / `StepInfo` contracts
* Per-episode metadata: body_poses, contact_events, is_success, reward
  breakdown
* Provenance: prompt version, LLM model, sim seed, validation report

**Consumers:**
* `McapEpisodeLoader` / `EpisodeDataset` → behavior cloning
* `ClankerGoalEnv` + HER replay → goal-conditioned offline RL
* `evaluate_policy()` → zero-shot success rate measurement
* Warm-start for online RL via `ClankerGymnasiumEnv`

**Non-goals:**
* Real robot control (simulation-first)
* Replacing RL end-to-end (this bootstraps; RL refines)
* Image-based planning (LLM operates on structured state, not pixels)

#### 4.1 SceneSpec — **[NEW]** Python Pydantic model, serialized to JSON

Wraps existing Rust configs into a single document the LLM can consume.

```python
from pydantic import BaseModel, Field

class SimulationSpec(BaseModel):
    gravity: list[float] = Field(default_factory=lambda: [0.0, 0.0, -9.81])
    physics_dt: float = 0.001
    control_dt: float = 0.02
    max_episode_steps: int = 500
    seed: int = 42

class RobotSpec(BaseModel):
    name: str
    urdf_path: str
    base_position: list[float]
    base_orientation: list[float]
    fixed_base: bool = True
    control_mode: str = "position_pd"
    joint_names: list[str] = Field(default_factory=list)
    joint_limits: dict[str, list[float]] = Field(default_factory=dict)
    joint_types: dict[str, str] = Field(default_factory=dict)
    ee_link_name: str = "end_effector"
    pd_gains: dict[str, list[float]] = Field(default_factory=dict)

class ObjectSpec(BaseModel):
    name: str
    shape: str
    shape_params: dict
    position: list[float]
    orientation: list[float] = Field(default_factory=lambda: [0, 0, 0, 1])
    mass: float = 0.1
    friction: float = 0.8
    restitution: float = 0.1
    is_static: bool = False
    is_graspable: bool = False
    is_container: bool = False

class ConstraintSpec(BaseModel):
    workspace_bounds_min: list[float]
    workspace_bounds_max: list[float]
    max_ee_speed: float = 1.0
    max_joint_velocity: float = 2.0
    max_contact_force: float = 100.0
    keep_upright: list[str] = Field(default_factory=list)
    avoid_regions: list[dict] = Field(default_factory=list)
    no_self_collision: bool = True

class ObservationSpec(BaseModel):
    modalities: list[str] = Field(default_factory=lambda: ["proprio"])
    joint_state_dim: int = 0
    ee_pose_dim: int = 7
    object_pose_dim: int = 0
    cameras: list[dict] = Field(default_factory=list)

class SceneSpec(BaseModel):
    scene_id: str
    units: dict = Field(default_factory=lambda: {"length": "m", "angle": "rad", "time": "s"})
    simulation: SimulationSpec
    robot: RobotSpec
    objects: list[ObjectSpec]
    constraints: ConstraintSpec
    observations: ObservationSpec
```

#### 4.2 TaskSpec — **[NEW]** Python Pydantic model

```python
class TaskSpec(BaseModel):
    """Task description for LLM planning."""
    task_id: str
    task_text: str       # Natural language: "Pick up the cube and place it in the bin."

    # Structured success criteria (maps to TerminationFn on Python side)
    success_criteria: list["SuccessCriterion"]

    # Reward hint for the LLM (actual reward is Python-side RewardFunction)
    reward_hint: str = ""   # e.g. "Dense distance EE→object + sparse success bonus"

    # Preferences (soft constraints)
    preferences: dict = Field(default_factory=dict)
    # Examples:
    #   "minimize_time": true
    #   "prefer_top_grasp": true
    #   "gentle_placement": true


class SuccessCriterion(BaseModel):
    """One success condition. Maps to Python TerminationFn."""
    type: str
    params: dict
    # Types:
    #   "object_near_target"  → {"object": "cube", "target": [x,y,z], "threshold": 0.05}
    #   "object_in_container" → {"object": "cube", "container": "bin"}
    #   "object_upright"      → {"object": "bin", "max_tilt_rad": 0.2}
    #   "ee_near_target"      → {"target": [x,y,z], "threshold": 0.05}
    #   "contact_between"     → {"body_a": "gripper", "body_b": "cube", "min_force": 1.0}
```

### 5) LLM Output Format — Skill Plan

The LLM outputs a **Parameterized Skill Plan**: a sequence of motion primitives
that the `SkillCompiler` converts to joint-space trajectories via the existing
DLS IK solver and position PD control.

#### 5.1 Skill Plan Schema (LLM output)

```json

  "plan_type": "skill_plan",
  "rationale": "Top-down approach to cube, close gripper, lift, move over bin, lower, release.",

    "Cube is reachable from above within arm workspace",
    "Top-down grasp is feasible (no obstructions above cube)",
    "Bin opening is wide enough for cube + gripper clearance"

    }

```

#### 5.2 Skill Vocabulary (Exhaustive)

Every skill the LLM may emit.  The `SkillCompiler` rejects unknown names.

| Skill | Params | Execution |
|-------|--------|-----------|
| `move_to` | `target: {frame, position}`, `orientation?`, `speed_fraction`, `tolerance?` | IK solve target → interpolate joint positions at `speed_fraction * max_joint_vel` → PD track via `MotorOverrides` |
| `move_linear` | `direction: [x,y,z]`, `distance: m`, `speed_fraction`, `guard?` | Cartesian straight-line: current EE + direction*distance. Step-wise IK. Optional early stop on `guard` (contact force or distance). |
| `move_relative` | `frame: "world"\|"ee"\|<object_name>`, `delta: [dx,dy,dz]`, `speed_fraction` | Move EE by delta relative to reference frame. IK at each waypoint. |
| `set_gripper` | `width: m`, `force?: N`, `wait_settle_steps?: int` | Set prismatic gripper joint target. Wait for settle or contact. |
| `wait` | `steps: int` | Hold current joint positions for N control steps. |
| `move_joints` | `targets: {joint_name: rad, ...}`, `speed_fraction` | Direct joint-space interpolation, bypassing IK. For known configurations (home, stow). |

**Reference frames:**
* `"world"` — Rapier physics frame (Z-up, meters)
* `"ee"` — current end-effector frame
* `<object_name>` — object's current pose from `StepInfo.body_poses[name]`

**Guard conditions** (optional early termination for a skill):
* `{"type": "contact", "body": "<name>", "min_force": N}` — stop when `ContactEvent` with body exceeds threshold
* `{"type": "distance", "from": "<name>", "threshold": m}` — stop when EE is within distance of object
* `{"type": "timeout", "steps": int}` — max steps for this skill

#### 5.3 LLM Output Constraints

The system prompt instructs the LLM to:

1. **Output strict JSON** matching the schema above — no markdown, no commentary outside `"comment"` fields
2. **Use only declared skill names** from the vocabulary table
3. **Reference only objects present in the SceneSpec** by their `name` field
4. **Stay within workspace bounds** — all target positions must satisfy `constraints.workspace_bounds`
5. **Respect joint limits** — the skill compiler validates IK solutions against URDF limits
6. **Include `assumptions`** — explicit list of environmental preconditions the plan depends on
7. **Include `uncertainty_flags`** — anything the LLM is uncertain about (e.g., "grasp orientation may need adjustment")
8. **Keep plans short** — prefer 4-10 skills per plan; longer plans have higher failure rates

#### 6.1 `SceneSpecBuilder` — **[NEW]** Python

**Purpose:** Construct `SceneSpec` from existing Clankers configs.

* **In:** URDF path, `SimConfig` dict, object placement list, constraint config

* **Implementation:** Reads URDF to extract joint names/limits/types, combines

```python
class SceneSpecBuilder:
```

#### 6.4 `PlanParser` — **[NEW]** Python

**Purpose:** Validate and normalize LLM output.

* **In:** raw `LLMProposedPlan` JSON, `SceneSpec`
* **Out:** `CanonicalPlan` with resolved references, or rejection with reasons
* **Checks:**
  - JSON schema validation
  - All skill names in vocabulary
  - All object references exist in `SceneSpec.objects`
  - All target positions within workspace bounds
  - Quaternions are unit-norm (normalize if close, reject if not)
  - speed_fraction in [0.0, 1.0]
  - gripper width in [0.0, max_gripper_width]

```python
from pydantic import BaseModel
from typing import Optional

class CanonicalPlan(BaseModel):
    plan_id: str
    skills: list["ResolvedSkill"]    # Object refs resolved to world-frame positions
    assumptions: list[str]
    metadata: dict                 # LLM model, prompt version, raw response hash

class ResolvedSkill(BaseModel):
    name: str
    target_world_position: Optional[list[float]] = None   # Resolved to world frame
    target_orientation: Optional[list[float]] = None       # Quaternion [qx,qy,qz,qw]
    params: dict                                 # All other parameters
    guard: Optional[dict] = None

class PlanParser:
    def parse(self, raw: dict, scene: SceneSpec) -> CanonicalPlan | PlanRejection: ...
```

#### 6.5 `SkillCompiler` — **[NEW]** Python (calls Rust engine via TCP)

**Purpose:** Convert `CanonicalPlan` into a sequence of joint-space actions that
the Clankers engine can execute via the standard `step(action)` protocol.

* **In:** `CanonicalPlan`, connected `ClankerGymnasiumEnv` (provides IK + physics)
* **Out:** `list[np.ndarray]` — sequence of actions in normalized [-1, 1] space

**Implementation approach:**
The compiler does NOT pre-compute the full trajectory offline.  Instead, it
drives the simulation step-by-step, using `StepInfo.body_poses` for real-time
EE position feedback and IK solving.

```python
class SkillCompiler:
    """Execute a CanonicalPlan through a live ClankerGymnasiumEnv."""

    def __init__(
        self,
        env: ClankerGymnasiumEnv,
        ik_max_iterations: int = 100,
        ik_tolerance: float = 1e-4,
    ) -> None: ...

    def execute(self, plan: CanonicalPlan) -> ExecutionTrace:
        """
        Step through the plan skill-by-skill.

        For each skill:
        1. Compute target joint positions (IK for Cartesian skills,
           direct for joint-space skills)
        2. Interpolate from current to target over N steps.
           Ensure division by zero is avoided if speed_fraction is 0.
           N = distance / max(1e-6, speed_fraction * max_vel * control_dt)
        3. env.step(action) for each interpolated waypoint
        4. Check guard conditions via StepInfo.contact_events / body_poses
        5. Record (obs, action, next_obs, info) into trace

        Returns ExecutionTrace on completion (success or failure).
        """
        ...
```

**IK integration:**
The SkillCompiler uses the Python-side IK from the arm_setup pattern:
- `DlsSolver` with `max_iterations=100, tolerance=1e-4, damping=0.01`
- Joint limits from URDF (parsed in `SceneSpec.robot.joint_limits`)
- Falls back to closest reachable if target is at workspace boundary

**Action format:**
Actions are `np.float32` arrays normalized to [-1, 1], where each element is:
```
action[i] = (target_position[i] - joint_center[i]) / joint_half_range[i]
```
This matches the existing `Action::Continuous` / `Action.clip_normalized()` convention.

#### 6.6 `SimValidator` — **[NEW]** Python

**Purpose:** Validate an executed trace against success criteria and constraints.

* **In:** `ExecutionTrace`, `TaskSpec`, `ConstraintSpec`
* **Out:** `ValidationReport`

```python
from pydantic import BaseModel
from typing import Optional

class ValidationMetrics(BaseModel):
    total_steps: int
    max_contact_force: float     # From StepInfo.contact_events
    max_joint_velocity: float    # From observations
    max_ee_speed: float          # Computed from body_poses deltas
    final_object_poses: dict[str, list[float]]  # From StepInfo.body_poses
    success_at_step: Optional[int] = None  # First step where all criteria met

class ConstraintViolation(BaseModel):
    type: str                    # "workspace_bounds" | "max_force" | "collision" | ...
    step: int
    details: str

class ValidationReport(BaseModel):
    passed: bool
    task_success: bool           # All SuccessCriteria met at final step
    constraint_violations: list[ConstraintViolation]
    metrics: ValidationMetrics
    failure_reason: Optional[str] = None   # Human-readable summary for PVCB refinement

class SimValidator:
    def validate(
        self,
        trace: ExecutionTrace,
        task: TaskSpec,
        constraints: ConstraintSpec,
    ) -> ValidationReport: ...
```

**Validation checks (using existing engine data):**

| Check | Data Source | Gate |
|-------|------------|------|
| Task success | `StepInfo.is_success` or Python `TerminationFn` | Hard |
| Workspace bounds | `StepInfo.body_poses[ee_link]` each step | Hard |
| Max contact force | `StepInfo.contact_events[].force_magnitude` | Hard |
| Joint limits | Observation vector (joint positions) vs URDF limits | Hard |
| Max EE velocity | Finite difference on `body_poses[ee_link]` / dt | Soft (warn) |
| Object stability | `body_poses[keep_upright_objects]` final orientation | Hard if specified |
| Avoid regions | `body_poses` vs `constraints.avoid_regions` | Hard if specified |

#### 6.7 `PVCBRefiner` — **[NEW]** Python

**Purpose:** Candidate-level refinement inside PVCB (not a standalone stage).

* **In:** `CanonicalPlan`, `ValidationReport`, `SceneSpec`, `TaskSpec`
* **Out:** `RefinedPlan` or candidate rejection
* **Strategy:**
  1. Deterministic fixes first (e.g., reduce speed_fraction if max_force exceeded,
     increase clearance if workspace bounds violated)
  2. Optional LLM re-proposal with failure context if deterministic fix fails

```python
class PVCBRefiner:
    def __init__(
        self,
        planner: LLMPlanner,
        parser: PlanParser,
        max_iterations: int = 3,
    ) -> None: ...

    def refine(
        self,
        plan: CanonicalPlan,
        report: ValidationReport,
        scene: SceneSpec,
        task: TaskSpec,
    ) -> CanonicalPlan | None:
        """Returns refined plan or None if candidate is rejected."""
        ...
```

**Refinement prompt addition:**

```
## Previous Plan Failed

{failure_reason}

## Constraints Violated
{constraint_violations_list}

## Attempt History
{attempt_history_list}

Revise the candidate to avoid these failures and previous incorrect attempts. Output the complete revised plan.
```

#### 6.8 `TrajectoryAugmenter` — Python (uses existing `DomainRandConfig`)

**Purpose:** Generate augmented variants of validated trajectories.

* **In:** validated `ExecutionTrace`, `SceneSpec`, augmentation config
* **Out:** list of augmented `ExecutionTrace` (re-validated in sim)

**Augmentation strategies using existing Clankers systems:**

| Strategy | Clankers Feature | Implementation |
|----------|------------------|----------------|
| Mass perturbation | `MassRandomizer` | Re-execute plan with randomized link masses |
| Friction perturbation | `SurfaceFrictionRandomizer` | Re-execute with randomized friction coefficients |
| Actuator noise | `ActuatorRandomizer` | Randomize motor/transmission/friction params |
| Geometry scale | `GeometryRandomizer` | Randomize collider half-extents |
| Object pose jitter | `ObjectConfig.position` | Perturb initial object positions ±5cm, re-execute |
| Initial joint noise | `RobotConfig.initial_joint_positions` | Small random offsets to starting config |
| External force | `ExternalForceRandomizer` | Random perturbation forces during execution |

**Key: every augmented trajectory is re-executed through the Rapier3D physics
and re-validated.**  We do NOT apply geometric transforms to recorded data —
we replay the plan with modified physics parameters and verify the result.

```python
class TrajectoryAugmenter:
    def __init__(
        self,
        n_augmentations: int = 10,
        domain_rand_config: dict | None = None,
        object_jitter_range: float = 0.05,  # meters

        self,
        plan: CanonicalPlan,

        task: TaskSpec,
        base_seed: int = 0,
    ) -> list[tuple[ExecutionTrace, ValidationReport]]:
        """
        For each augmentation seed:
        1. Perturb SceneSpec (object poses, physics params)
        2. Execute plan in fresh sim instance
        3. Validate result
        4. Keep only passing traces
        """
        ...
```

#### 6.10 `Pretrainer` — Python

**Purpose:** Train initial policy from synthetic dataset.

* **In:** `EpisodeDataset`, model config
* **Out:** pretrained policy weights, `EvalResult`

**Training modes:**

1. **Behavior Cloning (BC):** MSE loss on `||π(obs) → action||` with action
   noise regularization.  Uses `EpisodeDataset` directly.

2. **Goal-Conditioned BC:** Uses `ClankerGoalEnv` observation format.
   Dataset episodes are relabeled with achieved goals (from final
   `StepInfo.body_poses`) for HER-style training.

3. **Offline RL Warm-Start:** Load episodes into SB3 replay buffer via
   `McapEpisodeLoader.to_sb3_replay_buffer()`, then run SAC/TD3 with
   pretrained weights.

**Evaluation:**
```python
# After pretraining:
result = evaluate_policy(env, policy, n_episodes=100, reward_fn=composite_reward)
print(f"Zero-shot success: {result.success_rate:.1%}")
print(f"Mean reward: {result.mean_reward:.3f}")
for name, value in result.reward_breakdown.items():
    print(f"  {name}: {value:.3f}")
```

### 7) Validation Requirements (Hard Gates)

LLM propositions **never** become training data until validated in Rapier3D.

**Minimum validation checks:**

| Check | Source | Gate Type |
|-------|--------|-----------|
| IK solvable for all Cartesian targets | DLS solver convergence | Hard reject |
| Joint limits satisfied at every step | Observation vs URDF limits | Hard reject |
| No workspace boundary violations | `StepInfo.body_poses[ee]` vs `ConstraintSpec.workspace_bounds` | Hard reject |
| Contact forces within limits | `StepInfo.contact_events[].force_magnitude` | Hard reject |
| Task success achieved | `StepInfo.is_success` or Python `TerminationFn` at final step | Hard reject |
| Object stability maintained | `StepInfo.body_poses[keep_upright]` orientation check | Hard if specified |
| Avoid-region compliance | `body_poses` vs `ConstraintSpec.avoid_regions` | Hard if specified |
| Trajectory smoothness | Finite-difference jerk on joint positions | Soft (log warning) |
| Episode length reasonable | `StepInfo.episode_length` < `max_episode_steps` | Soft (log warning) |

**Validator output is structured** — failure reasons are machine-readable
so PVCB refinement can generate targeted candidate updates.

### 8) Execution Contracts & Protocol

#### 8.1 Clankers TCP Protocol (Existing)

The datagen module communicates with the Rust engine via the existing
length-prefixed JSON TCP protocol:

| Message | Direction | Purpose |
|---------|-----------|---------|
| `Init` | Python → Rust | Handshake, negotiate capabilities, get `EnvInfo` |
| `Reset` | Python → Rust | Reset episode, get initial observation |
| `Step {action}` | Python → Rust | Execute action, get `StepResult` + `StepInfo` |
| `Close` | Python → Rust | Teardown connection |
| `BatchReset` | Python → Rust | Parallel env reset (for augmentation workers) |
| `BatchStep` | Python → Rust | Parallel env step |

**Frame format:** 4-byte LE u32 length + UTF-8 JSON payload (max 16 MiB)

**Key `StepInfo` fields for validation:**
```json
{
  "episode_length": 42,
  "body_poses": {
    "end_effector": [0.2, 0.0, 0.92, 0.0, 0.707, 0.0, 0.707],

  "contact_events": [
    {"body_a": "gripper_finger_l", "body_b": "cube", "force_magnitude": 12.5}

}
```

### 11) Safety & Guardrails

* **LLM output is data, not code.** Plans are JSON skill sequences — no
  arbitrary code execution.
* **All positions bounds-checked** by `PlanParser` against `ConstraintSpec`
  before any simulation execution.
* **Action clipping** by `ClankerGymnasiumEnv.step()` — continuous actions
  are clipped to `action_space.low`/`action_space.high` before sending to
  the Rust engine (already implemented).
* **Max contact force gate** — episodes with forces exceeding
  `constraints.max_contact_force` are rejected.
* **Deterministic execution** — `SimConfig.deterministic_mode` ensures
  reproducible rollouts for debugging.
* **No direct LLM → simulator execution.** The pipeline is:
  `LLM → JSON → PlanParser (validate) → SkillCompiler (step-by-step) →
  SimValidator (post-hoc check)`.

### 15) Missing Component Spec (Python-First, OpenAI Backend)

#### 15.1 Execution boundary (explicit)

This datagen system is **Python-land orchestration**.

* **Python owns:** planning, prompting, JSON parsing, skill compilation logic,
  validation logic, PVCB candidate refinement, augmentation orchestration, dataset packaging,
  training handoff, CLI/tools.
* **Rust owns only:** physics stepping, simulation state, rendering/viz, and
  core environment protocol/runtime.
* No Rust-side LLM integration is required for this feature.
* Packaging rule: synthetic generation lives in a standalone Python module,
  not inside gym wrappers.

#### 15.2 Required new Python package

Create a new standalone package:

```

  compiler.py           # skill-by-skill executor over ClankerGymnasiumEnv
  validator.py          # hard/soft gate checks -> ValidationReport
  pvcb_refiner.py       # deterministic candidate refinement + optional re-prompt
  augment.py            # physics-domain-rand replay orchestration
  packager.py           # MCAP/provenance/splits/manifest writer
  pipeline.py           # end-to-end generate_dataset entry point
  cli.py                # command-line entrypoint
```

#### 15.3 OpenAI planner spec (assumed backend)

`OpenAIPlanner` (new, Python):

* Uses OpenAI Responses API (JSON-only output mode).
* Inputs:
  * `scene: SceneSpec`
  * `task: TaskSpec`

#### 15.6 Validator spec (hard gates)

`SimValidator` (new, Python):

* Input: `ExecutionTrace`, `TaskSpec`, `ConstraintSpec`
* Output: `ValidationReport`

  * workspace violation
  * joint limit violation
  * max contact force violation
* Soft warnings:
  * EE speed over preferred threshold
  * excessive jerk / trajectory roughness
* Must emit:
  * `failure_reason` string
  * structured `violations[]` records

* Output:
  * refined `CanonicalPlan` or rejection record

* **Schema style:** use one model style (Pydantic) for datagen contracts.
* **CI behavior:** no live LLM calls in default CI/CD; only mocked planner tests.
* **Live LLM validation:** optional, opt-in, and excluded from normal CI.

### 17) Force Estimation Note (Simulation Contact Forces)

Force-based validation is relevant and should remain in scope, with the
following implementation note:

* Current force signal is solver-derived in Rust physics:
  * contact force is computed as `total_impulse / dt` from Rapier narrow-phase
    contacts.
* This is physically meaningful for gating, but has current integration gaps:
  * `StepInfo.contact_events` is currently not populated by gym step results
    (step info mostly carries `episode_length` + defaults).
  * contact mapping is currently strongest for robot-linked bodies; scene
    object coverage in exported step info is incomplete.
* Spec requirement for datagen implementation:
  * add a step-info enrichment path that emits `contact_events` with
    `{body_a, body_b, force_magnitude}` every step.
  * ensure object and robot body names are both mapped so force gates can be
    evaluated uniformly.
* Validator policy until enrichment lands:
  * treat max-contact-force as a provisional gate only when contact events are
    present; otherwise record a warning and do not silently claim force checks
    passed.

### 18) Multi-Interpretation Candidate Sampling and Selection

**Proposed name:** `Physics-Validated Candidate Bank (PVCB)`

Purpose:
* Generate multiple plausible interpretations of the same task (different
  motion styles, speed/force profiles, approach paths), then keep only
  trajectories that pass simulation validation.

How it works:
1. For each `(SceneSpec, TaskSpec)`, sample `K` LLM candidates with controlled
   diversity (temperature, seed, and optional style hints such as
   `gentle_contact`, `fast_transfer`, `conservative_clearance`).
2. Optionally apply deterministic parameter sweeps per candidate (for example
   speed fractions, settle steps, grip force ranges) to create `K x M`
   executable plan variants.
3. Execute all variants in simulation.
4. For failing near-miss candidates, run `PVCBRefiner` (deterministic edits,
   then optional re-proposal) and re-execute within per-candidate attempt caps.
5. Score each execution with `ValidationReport` gates and metrics.
6. Keep only passing variants for training; reject all failing ones.

Selection policy:
* Hard filter: all safety/task gates must pass.
* Ranking among passers: prefer lower contact-force peaks, smoother motion,
  shorter episodes, and higher task margin.
* Retain top `N` per scene/task or sample from passers to preserve diversity.

Why this reduces force-signal misspec risk:
* Plans are selected by observed simulation outcomes, not by LLM intent text.
* Even if some candidate plans imply poor force behavior, those are discarded
  by validation; only empirically valid trajectories enter training.

Required metadata additions:
* `candidate_group_id`, `candidate_index`, `variant_index`
* `selection_rank`, `selection_reason`
* full per-candidate validation summary (including rejected candidates)

### 19) Operational Guardrails (Implementation Requirements)

1. Interface boundary contract:
* Define and freeze the `clankers_synthetic` -> `clankers` integration surface
  (env calls, expected `info` keys, seed/reset semantics).
* Treat any additional cross-package dependency as an explicit reviewed change.

2. Versioned dataset schema:
* Version `metadata.json`, provenance payloads, and episode annotations with
  `schema_version`.
* Compatibility policy:
  * major version = breaking
  * minor version = additive/non-breaking

3. Deterministic replay contract:
* Provide replay by `episode_id` with fixed seeds and plan hash.
* Replay must verify hash-equivalent summary metrics (or fail with structured
  mismatch report).

4. Failure taxonomy normalization:
* Use stable machine-readable failure codes (parser, IK, constraints, task, io).
* Keep free-text messages as secondary diagnostic fields only.

5. Separate gates from ranking:
* Hard gates determine admissibility into dataset.
* Ranking signals order/select among admissible candidates only.
* Do not allow ranking improvements to bypass failed hard gates.

6. Resource budget controls:
* Configure hard caps for:
  * candidates per scene/task
  * refinement attempts per candidate
  * wall-clock budget per scene/task
  * token budget per scene/task
* Exceeding caps must return structured partial-result status, not hang.

7. Force-gate evaluability status:
* Record force-gate status per episode as:
  * `passed` | `failed` | `not_evaluable`
* Do not collapse `not_evaluable` into `passed`.

8. CI no-network enforcement:
* Default CI test jobs must block outbound network access.
* OpenAI/planner tests must run via mocks/fixtures only.
* Live API tests must be opt-in (manual/nightly) and excluded from required CI.
