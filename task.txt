# 1) Workstream A ‚Äî Eliminate long-run drift in simulation (engine-agnostic + practical)

Long-run drift is usually cumulative constraint/contact error + controller injecting tiny biases that integrate. You need *both* simulation tuning and controller bias rejection.

## A1. Switch to multi-rate stepping (biggest sim stability win per effort)

You already saw dt matters: **0.01 s (100 Hz) is essential for 1 m/s**. 
Now do:

* **physics_dt = 0.0025‚Äì0.005 s** (200‚Äì400 Hz physics)
* **control_dt = 0.01 s** (100 Hz control) *or* 0.005 s (200 Hz control) once stable
* Apply motor commands at control_dt; hold them constant across substeps.

This matches established robotics sim guidance: effective timestep is `dt/substeps` and shorter timesteps improve stability for articulated assemblies. ([Robotsfan Docs][1])
PhysX best practices explicitly recommend using smaller time steps / multiple simulation calls rather than cranking solver iterations alone. ([Documentation Help][2])

**Implementation detail**

* In `clankers-physics`, add `substeps: u32` and step loop:

  * for _ in 0..substeps: integrate(physics_dt); solve_constraints();
* Expose `--physics-dt`, `--substeps` in `quadruped_mpc_bench`.

## A2. Add compliance / soft constraints at contacts and/or joints

Engines that are iterative constraint solvers behave better when constraints aren‚Äôt infinitely stiff. Robotics simulators widely expose this as ERP/CFM or ‚Äúsolver impedance‚Äù.

* In ODE/Gazebo, **CFM stabilizes intermittent contacts by softening constraints**; the Gazebo tutorial even calls it ‚Äúfirst parameter to tune‚Äù for unstable sims. ([Gazebo][3])
* MuJoCo exposes **constraint impedance and multiple solver algorithms** (PGS/CG/Newton) and explicitly ties stability to solver parameters + timestep. ([MuJoCo Documentation][4])

**Implementation approach (engine-agnostic)**

* If your engine supports *any* contact softness / compliance / penetration correction tuning, do:

  * Slightly soften normal constraint (avoid perfectly rigid contacts)
  * Add a small amount of tangential damping to reduce micro-slip creep
* If not directly exposed, emulate via:

  * Adding small vertical compliance at the foot (virtual spring-damper in WBC) when in stance
  * Keeping friction consistent with the MPC cone (see A4)

## A3. Reduce joint/constraint drift with reduced-coordinate articulations (optional but high leverage)

Maximal-coordinate ‚Äúrigid bodies + joints‚Äù can accumulate joint error over long horizons. Reduced-coordinate articulations largely eliminate this class of drift.

* PhysX articulations are **reduced-coordinate** and described as having ‚Äúzero joint error by design.‚Äù ([NVIDIA Omniverse][5])
* NVIDIA also provides a dedicated robot stability guide emphasizing timestep reduction and compliance. ([NVIDIA Omniverse Documentation][6])

**Decision point**

* If you need **10+ minutes drift-free** and Rapier-style joints keep accumulating error, plan a backend swap to either:

  * **MuJoCo** (double precision by default, robotics-oriented constraints) ([MuJoCo Documentation][7])
  * **PhysX articulations** (Isaac Sim / Isaac Lab) ([NVIDIA Omniverse][5])
    This can be done behind an interface so your MPC/WBC code stays intact.

## A4. Make friction consistent (stop slow ‚Äústance creep‚Äù)

Your ground friction is **1.0** in sim, but MPC uses **Œº=0.4**.  
That mismatch can produce weird regimes where the controller expects slip sooner/later than the engine actually allows, and tiny slip integrates into drift.

**Rule**: set sim friction ‚âà MPC Œº for baseline runs; then test robustness with ¬±30% Œº domain randomization.

**Implementation**

* `--mu-sim`, `--mu-mpc` CLI flags
* Add metric: `slip_speed = ||v_foot_tangential||` during stance

## A5. Handle floating-point precision over long travel

If any part of your sim pipeline is `f32`, drift can explode as the robot walks away from origin. Two options:

* **Double precision physics** (MuJoCo uses double by default). ([MuJoCo Documentation][7])
* **Floating origin / world shifting**: periodically re-center the world so the robot stayie for large-world physics stability. ([Godot Engine documentation][8])

**Implementation**

* Every N meters traveled, subtract robot base pose from all bodies (world shift) and accumulate an ‚Äúabsolute world transform‚Äù separately for logging.

# 2) Workstream B ‚Äî Make gait scheduling adaptive (simple + modular + efficient)

Right now trot is duty=0.5, cycle=0.35 s ‚Üí stance=0.175 s.

### B1.1 Feasibility-based timing law (deterministic, cheap)

Derive cycle time from max feasible step length:

* Approx symmetric step length: `L ‚âà v * T_stance`
* Constraint: `L ‚â§ L_max_forward` (choose 0.20‚Äì0.25 m first; don‚Äôt use 0.30 m at the limit)

So:

* `T_stance ‚â§ L_max / v`
* `T_cycle = T_stance / duty`

Also nstraint:

* `T_swing ‚â• T_swing_min` (from foot clearance + jointlduty)`

**Implementation**

* Add config:

  * `L_max = 0.22 m` (start conservative)
  * `T_swing_min = 0.12‚Äì0.16 s` (based on your 0.10 m step height) 
* Compute:

  * `T_cycle = clamp(Tmin, Tmax, min(L_max/(v*duty), T_swing_min/(1-duty)))`
* Make `duty` a function of speed too (tends to decrease slightly at higher speeds).

### B1.2 Multi-gait selection by ‚Äúone-step lookahead scoring‚Äù

At low frequency (e.g., 10‚Äì20 Hz), evaluate a small candidate set:

* {walk, trot, bound} √ó {2‚Äì3 duty values} √ó {2‚Äì3 cycle values} = ~12‚Äì27 candidates

For each candidate:

1. Build contact schedule for horizon
2. Run MPC once
3. Score:

   * QP objective
   * max friction utilization `max ||f_t||/(Œº f_n)`
   * max normal utilization `max f_n/f_max`
   * predicted slip proxy (see C2)
   * tracking error

Pick best + hysteresis (avoid chatter).

This is still simple, and computationally feasible given your QP is ~0.1‚Äì0.5 ms. 

### (Newest research tie-in)

* Recent work explicitly studies **real-time gait adaptation in a continuous gait space** (optimizing gait variables online) but uses hing. ([arXiv][9])
  Your candidate-scoring approach is the ‚Äúdeterministic MPC engineer‚Äù version of the same idea.

## B2. Phase-lock gait timing to real contacts (kills schedule drift)

You currently advance phase purely by time. 
If touchdown/liftoff timing slips even slightly (contact solver jitter, micro slip, etc.), the controller and schedule de-sync over tens of seconds ‚Üí ‚Äúslow drift‚Äù ‚Üí tip.

**Add a Phase-Locked Loop (PLL) on contact events**

* Detect touchdown event for each foot: `contact && prev_no_contact`
* Maintain an estimate of phase error:

  * `e = phase_now - expected_touchdown_phase(foot)`
* Apply a small correction to phase rate:

  * `phase_dot = 1/T_cycle - k_pll * e`
* Clamp correction to avoid instability.
  erride‚Äù**
* If gait says stance but contact is absent (unloaded), treat as swing in MPC constraints for next solve.
* If early contact happens during swing, allow stance forces early (with small ramp).

This is extremely effective and cheap.

# 4) Workstream D ‚Äî MPC formulation upgrades (still convex, but much more robust)

## D1. Ensure horizon covers at least one contact transition

Your preview is `H*dt = 0.1‚Äì0.2 s` depending on dt/horizon. 
But trot swing/stance are **0.175 s** each. 
If the horizon does not include the upcoming contact switch, MPC can ‚Äúpaint itself into a corner‚Äù at higher speed.

**Rule**: `T_horizon >= 1.2 * max(T_swing, T_stance)`

Implementation:

* If control_dt=0.01, set `H=20‚Äì25` (0.2‚Äì0.25 s)
* set `H=30‚Äì40` (0.15‚Äì0.2 s)

## D2. Add force rate regularization (prevents contact jitter)

Add penalty on `Œîf`:

* `Œ£ ||f_k - f_{k-1}||¬≤`

This reduces high-frequency force chatter that excites contact solvers.

## D3. Add soft constraints (slack vars) for friction cone & f_n bounds

QP infeasibility and solver ‚Äúboundary weirdness‚Äù often shows up as long-run drift.

Add slack:

* friction: `||f_t|| ‚â§ Œº f_n + s_fric`, `s_fric ‚â• 0`
* normal: `0 ‚â§ f_n ‚â§ f_max + s_n`, `s_n ‚â• 0`
  Penalize `s` heavily in cost.

This keeps the solver always returning a meaningful solution while telling you when feasibility is tight (log slack!).

## D4. Add disturbance/bias states to kill ‚Äúslowbust centroidal MPC work shows formal stabilityive reformulations and handles constant disturbances. ([arXiv][10])

You can implement a simpler ‚Äúengineering version‚Äù:

Augment centroidal dynamics with constant acceleration bias:

* `vÃá = (1/m) Œ£ f_i + [0,0,-g] + d_v`
* `œâÃá = I^{-1} Œ£ (r√óf) + d_œâ`
  with `d_v, d_œâ` treated as **random-walk / constant over horizon** states.

Estimate `d` online from measured vs predicted acceleration (simple EKF or even a low-pass ‚Äúinnovation integrator‚Äù).

This dramatically reduces drift from:

* slightly wrong mass/inertia
* slight contact model mismatch
* solver creep causing tiny unmodeled impulses

**If you want a reference implementation**

* The stable centroidal MPC paper has open-source code (humanoid + quadruped modules). ([GitHub][11])

## D5. Upgrade yaw-only inertia rotation to full orientation linearization (optional)

Yaw-only inertia rotation is a good convexity trick, but at higher speeds roll/pitch excursions matter. 

Keep QP convex by:

* computing full `R(roll,pitch,yaw)` from current state
* freezing `R` over the horizon (time-varying linear MPC)
* dynamics remain linear in forces for that tick

This is a ‚Äúdrop-in accuracy upgrade‚Äù without moving to NMPC.

# 5) Workstream E ‚Äî Solver & performance engineering (so you can afford adaptation)

## E1. Make QP data updates allocation-free

Your condensed matrices likely share a fixed sparsity pattern for a given horizon/contact topology. Use:

* preallocated CSC buffers
* overwrite numeric values in-place
* keep solver object alive

Caveat: interior-point solvers re-factor frequently; ‚Äúparametric warm start‚Äù often yields less benefit than with ADMM/active-set solvers. Clarabel‚Äôs own discussion notes that repeatedly updating data in-place isn‚Äôt as meaningful as in warm-start-friendly methods. ([GitHub][12])

## E2. Evaluate an MPC-friendly warm-start solver (optional)

If you start evaluating multiple candidate gaits per tick, warm start becomes valuable.

Two robotics-focused options:

* **ProxSuite / ProxQP** supports warm-starting and cac([GitHub][13])
* **OSQP**-style operator splitting is popular for MPC due to predictable iteration costs (see recent solver surveys/benchmarks). ([arXiv][14])

Practical plan:

* keep Clarabel as baseline
* add `--solver={clarabel,proxqp,osqp}`
* benchmark ‚Äúsolve frequency per watt / per ms‚Äù on your exact condensed QP sizes (your repo already has a bench runner).

## F3. Mixed-integer foot powerful, but compute-heavy)

**Implementation reality**

* Good for rough terrain foothold selection; likely overkill for ‚Äúflat ground faster than 1 m/s.‚Äù

## F4. Learning-assisted continuous gait space adaptation (new, but code unclear)

* For each sample, run your convex QP once ‚Üí score ‚Üí pick
  This gets most benefits without Dreamer/world-model complexity.

## F5. Offline motion library + online MPC switching (modular & proven)

A mature architecture is: precompute motion primitives offline, then use online MPC to adapt/switch. ([crl.ethz.ch][18])

## Addendum: Repository Feasibility Review (2026-02-28)

This addendum summarizes feasibility after comparing `task.txt` against the current implementation in this repository. Existing content above is unchanged.

### Overall feasibility

- Milestones 1ñ3 are feasible in this codebase with moderate engineering effort.
- Milestones 4ñ5 and SOTA/research tracks are feasible but significantly higher effort and higher integration risk.

### What already aligns with the plan

- Fixed gait presets with duty/cycle/offsets and horizon contact matrix generation exist.
- Convex centroidal MPC with yaw-only inertia rotation is implemented.
- Clarabel-based condensed QP with contact/friction constraints is implemented.
- Multi-rate stepping support (physics substeps per control step) exists in core/physics layers.

### Main gaps versus the plan

- No adaptive gait supervisor yet (duty/cycle are preset or manually overridden, not closed-loop adaptive).
- No contact-phase locking (PLL) to measured touchdown/liftoff; phase progression is time-driven.
- In shared headless control path, swing torque is still paired with `target_pos = init_joint_angles`, which can conflict with swing Cartesian objectives.
- MPC robustness upgrades from this plan are not yet present:
  - force-rate regularization (`delta_f`) term,
  - slack variables for soft friction/normal constraints,
  - disturbance/bias state augmentation.
- No explicit slip detector feeding schedule/speed overrides.
- Horizon can be short at high control rates depending on runtime config (e.g., `H=10`, `dt=0.01` => 0.1 s preview).

### Practical conclusion

The current architecture is suitable for the roadmap in this document. The remaining work is primarily control integration, adaptation logic, and robustness extensions rather than a full rewrite.

Recommended implementation order for best ROI:

1. Remove swing command conflict in shared control path.
2. Add adaptive gait timing + contact-aware phase correction.
3. Add MPC robustness terms (`delta_f`, slack constraints), then disturbance/bias rejection.
4. Expand long-run validation (10-minute drift tests, friction randomization, push recovery).
